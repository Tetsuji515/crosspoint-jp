# Xteink X4 カスタムファームウェア 基本設計書

対象：crosspoint-reader-cjk ベースのカスタムファームウェア
関連文書：要件定義書（xteink_x4_firmware_requirements.md）

---

## 1. 全体アーキテクチャ方針

### 1.1 設計の基本原則

| 原則 | 内容 |
|---|---|
| 読書ファースト | 本機はあくまでe-reader。追加アプリは読書体験を妨げない位置づけとする |
| メモリ最優先 | ESP32-C3のRAMは約380KB。CrossPointの「SDカードへ積極的にキャッシュする」方針を全アプリで踏襲する |
| Activityベース | CrossPoint/biscuit系の画面遷移構造（onEnter/loop/render/onExitを持つActivity、ActivityManagerによるpush/pop管理）に準拠する |
| 無線の排他管理 | WiFiとBLEはアンテナ・無線リソースを共有するため、RadioManager的な一元管理層を設けて調停する |
| 段階的リリース | 機能ごとにブランチを分け、1機能ずつ実機確認しながらマージする |

### 1.2 レイヤー構成

```
┌─────────────────────────────────────────┐
│ Activities（画面）                        │
│  既存: Home / Reader / Settings / ...    │
│  追加: AppLauncher / Clock / Weather /    │
│        HabitTracker / Slideshow / Sudoku /│
│        MarkdownReader / ObsidianSync      │
├─────────────────────────────────────────┤
│ Services（共有ロジック）                   │
│  既存: EPUBパーサ / フォント / KOReader同期│
│  追加: MarkdownParser / VaultIndex /      │
│        SyncClient / WeatherClient /       │
│        AppDataStore                       │
├─────────────────────────────────────────┤
│ Platform（ハードウェア抽象）               │
│  GfxRenderer / SDカード / ボタン入力 /     │
│  RadioManager(WiFi・BLE調停) / RTC / 電源  │
└─────────────────────────────────────────┘
```

### 1.3 ホーム画面・メニュー構成（リスト型ランチャー）

**ホームUIを全面刷新する。** 本棚（書籍ライブラリ）も含めた全機能を「アプリ」として同列に扱い、物理ボタンでリストを回して選択するランチャーをホームとする（UIリニューアルの先行例としてinxフォークを参考にする）。

```
新ホーム = アプリランチャー（リスト/カルーセル型）
  ↕ [上][下]ボタンでリストを回す
├── 📚 本棚（既存の書籍ライブラリへ入る）
├── 📝 ノート（Markdown/Obsidianリーダー）
├── 🕐 時計・タイマー
├── ☀ 天気
├── ✓ 習慣トラッカー
├── 🖼 スライドショー
├── 🎲 数独
├── 🔄 Vault同期
├── 📤 ファイル転送（既存機能への導線）
└── ⚙ 設定（既存機能への導線）
```

**アプリレジストリ方式**：
- ホームはアプリ一覧データ（レジストリ）を読んで描画するだけの構造とする

```cpp
struct AppEntry {
  const char* nameKey;    // i18n文字列ID（例: "app.clock"）
  const uint8_t* icon;    // 1bitアイコン（例: 64×64）
  bool enabled;           // 実装済みフラグ
  Activity* (*create)();  // 画面生成関数（未実装ならnullptr）
};
```

- **未実装アプリもリストに最初から表示**し、「準備中」表記（グレーアウト相当の点線枠等）とする。実装完了後に `enabled = true` と生成関数の接続のみで有効化する
- これによりホームUIの開発と個別アプリの開発を完全に分離し、「ホームUI完成 → アプリを1つずつ実装・有効化」の段階的開発を可能にする
- 表示順はレジストリの並び順。将来的に並び替え設定も検討（初期スコープ外）
- **電源投入時の起動先も新ホームとする**が、「前回読んでいた本を直接開く（Quick Resume相当）」の既存挙動は設定で選択可能なまま残す（読書ファースト原則）

### 1.4 無線（WiFi/BLE）管理方針

| 機能 | 使用無線 | 使用タイミング |
|---|---|---|
| ファイル転送・KOReader同期（既存） | WiFi | ユーザー操作時のみ |
| 天気取得 | WiFi | 天気画面表示時・手動更新時 |
| Obsidian Vault同期 | WiFi | 手動実行時（将来: スケジュール実行） |
| NTP時刻同期 | WiFi | 時計初回設定時・手動更新時 |
| BLE機能（将来: ページめくりボタン等） | BLE | 該当機能の有効化時 |

**方針**：
- 無線は常時OFFを基本とし、必要な機能の実行時のみRadioManager経由でONにする（バッテリー・メモリ節約）
- WiFi使用中はBLEを起動しない（逆も同様）。排他制御はRadioManagerが担う
- WiFi接続情報（SSID/パスワード）は既存CrossPointの設定を共用する

### 1.5 データ保存方針（SDカード）

CrossPointの既存キャッシュディレクトリ（.crosspoint）と衝突しない専用領域を確保する。

```
SDカード
├── .crosspoint/        （既存キャッシュ、触らない）
├── books/              （書籍、既存運用）
├── fonts/              （フォント、既存運用）
├── images/slideshow/   ★スライドショー画像（PXC/BMP）
├── images/sleep/       ★スリープ画面（スクリーンセーバ）画像
├── vault/              ★Obsidian Vaultミラー
└── .apps/              ★アプリデータ
    ├── habits.json     （習慣トラッカー記録）
    ├── sudoku.json     （数独のセーブデータ）
    ├── weather.json    （天気キャッシュ）
    └── sync_state.json （Vault同期状態: ファイルごとのmtime記録）
```

- アプリデータはJSON形式（人が読めてデバッグしやすい・PCでも編集可能）
- 書き込みは変更時のみ・最小限に（SDカード寿命とレスポンス考慮）

### 1.6 電源・スリープとの関係

- 追加アプリ利用中も既存のスリープ（自動/電源ボタン）の仕組みに従う
- **ホームの時計はスリープ中は更新しない**。スリープ復帰時にその時点の時刻で時計エリアを再描画する（定期ウェイクアップによるスリープ中更新は行わない。バッテリー優先）
- **スリープ画面は任意の画像を表示するスクリーンセーバとする**。CrossPoint既存の「スリープ画面カスタム画像」機構を流用し、images/sleep/ フォルダの画像（PXC/BMP、パネル解像度に事前変換済み）を表示する
  - 単一固定画像 / スリープのたびにランダム or 順送りで切替、を設定で選択可能にする（切替はスリープ移行時の1回のみ描画するため電力影響なし）
  - スライドショーアプリ（2.4節）と画像フォーマット・変換フローを共通化する
- 時計・タイマーはスリープ復帰後に継続できるよう、開始時刻をSDに記録し経過時間を再計算する方式とする（RAM上のカウンタに依存しない）

## 2. 各アプリ 画面仕様

共通事項：
- 画面解像度：800×480（横）/ 480×800（縦）、1bitモノクロ
- 入力：物理ボタンのみ（前面4ボタン＋側面ボタン。以下、[上][下][決定][戻る]と表記し、実機のボタン配置確認後にマッピングを確定する）
- E-Inkの特性上、全画面リフレッシュは最小限にし、部分更新を基本とする

### 2.0 ホーム（リスト型ランチャー）★最初に開発

**画面構成**：上部1/4を時計エリア、残り3/4をアプリリストとする2分割レイアウト

```
┌────────────────────────────┐
│  12:34   7/11(土)    🔋87% │ ←時計エリア
│  （大型時刻＋日付表示）       │   (上部約1/4)
├────────────────────────────┤
│      🕐 時計・タイマー        │ ←前の項目(小)
│  ┌──────────────────────┐  │
│  │ 📚 本棚              │  │ ←選択中(大・反転)
│  └──────────────────────┘  │
│      📝 ノート(準備中)       │ ←次の項目(小)
│  ●○○○○○○○○○   1/10        │ ←位置インジケータ
└────────────────────────────┘
```

**時計エリアの仕様**：
- 高さ：縦持ち480×800なら上部約200px、横持ち800×480なら上部約120px
- 表示内容：現在時刻（大型フォント）＋日付・曜日＋バッテリー残量（従来のステータスバーを兼ねる）
- 更新：1分ごとに時計エリアのみ部分更新（リスト部は再描画しない）
- 時刻はRTC保持の値を表示。NTP未同期時は「--:--」＋「未同期」表示とし、時計・タイマーアプリ内から同期を促す
- 将来拡張：天気アプリ実装後、天気キャッシュ（weather.json）があればアイコン＋気温を時計横に表示するオプションを検討（ホームがミニダッシュボード化）

**操作**：
- [上][下]：リストを回す（端で反対側へループする）
- [決定]：選択中アプリを起動（`enabled = false` の項目は「準備中です」のトースト表示のみ）
- [戻る]：読書中の本があればそこへ復帰（Quick Resume連動）、なければ何もしない
- 長押し等の拡張操作は実機のボタン仕様確認後に検討

**表示仕様**：
- 選択中の項目は中央に大きく表示し、前後の項目を上下に小さく表示（カルーセル感の演出＋現在位置の把握性）
- ページ送りは部分更新で行い、残像が目立つ場合は N 回に1回全画面リフレッシュ
- 未実装（準備中）アプリは点線枠＋「準備中」ラベルで区別

**実装要点**：
- HomeActivity として新規実装し、起動時のエントリActivityを差し替える
- 既存の本棚画面（書籍一覧）は一切改変せず、「本棚」エントリからpushする構成にする（既存機能の安全確保）
- アプリレジストリ（1.3節）は静的配列で定義し、RAM消費を最小化する
- 時計エリアとリストエリアは独立した描画領域として実装し、それぞれ必要時のみ部分更新する（時計は1分ごと、リストはボタン操作時のみ）
- スリープ中は時計は更新されない。スリープ復帰時にホームへ戻った場合は時計エリアを再描画する

### 2.1 時計・タイマー

**画面構成**：タブ切替式の3画面（[上][下]でタブ移動）

```
┌────────────────────────────┐
│ [時計] タイマー ｽﾄｯﾌﾟｳｫｯﾁ    │ ←タブ
│                            │
│        12:34               │ ←大型表示
│     2026年7月11日(土)       │
│                            │
│  次回NTP同期: 起動時         │
└────────────────────────────┘
```

| 画面 | 機能 | 表示更新 |
|---|---|---|
| 時計 | 現在時刻・日付の大型表示 | 1分ごとに部分更新 |
| タイマー | プリセット（5/15/25/60分）＋カスタム。終了時は全画面点滅＋（可能なら）ブザー | 残り1分までは1分ごと、以降10秒ごと |
| ストップウォッチ | 開始/停止/リセット | 動作中は秒単位で部分更新 |

**データ・技術要点**：
- 時刻はNTP同期（WiFi）で取得しRTCに保持。同期は手動＋WiFi使用機会への相乗り
- ポモドーロ（25分/5分交互）はタイマーのプリセットとして実現（専用画面は作らない）
- 参考実装：biscuitのClock/Timer系アプリ

### 2.2 天気

**画面構成**：1画面完結

```
┌────────────────────────────┐
│ 東京            7/11(土)    │
│   ☀ 晴れ                   │
│   32℃ / 25℃   降水 10%     │
│ ────────────────────────── │
│ 明日: ☁ 30/24  明後日: ☂ 28/23│
│                            │
│ 最終更新 12:00  [決定]で更新  │
└────────────────────────────┘
```

**仕様**：
- 表示内容：現在天気・最高/最低気温・降水確率・3日分の予報
- 天気アイコンは1bitビットマップで自作（☀☁☂☃程度の数種類）
- 更新は「画面を開いた時にキャッシュが2時間以上古ければ自動取得」＋「[決定]で手動更新」
- 取得失敗時（WiFi未接続・PC不要のAPI直接取得失敗）はキャッシュ表示＋「オフライン」表示
- API：Open-Meteo（無料・キー不要・JSON）を第一候補とする
- 地点設定：設定画面で緯度経度 or 都市名を1箇所登録（複数地点は初期スコープ外）

### 2.3 習慣トラッカー

**画面構成**：2画面（一覧＋月間ビュー）

```
┌────────────────────────────┐
│ 今日の習慣        7/11(土)   │
│ [✓] 読書30分               │
│ [ ] 運動                   │
│ [✓] 早起き                 │
│                            │
│ [決定]ﾁｪｯｸ [戻る]長押:月表示  │
└────────────────────────────┘
```

**仕様**：
- 習慣は最大10件。登録・編集はデバイス上では最小限（名前入力が物理ボタンでは苦痛なため、habits.jsonをPCで編集する運用を正とし、デバイス上は「追加（プリセット名）・削除・並べ替え」のみ）
- 月間ビュー：カレンダー形式で達成日を●表示、連続日数（ストリーク）を表示
- データ：.apps/habits.json（習慣定義＋日付ごとの達成記録）
- 参考実装：biscuitのHabit Tracker

### 2.4 スライドショー

**画面構成**：全画面表示＋オーバーレイメニュー

**仕様**：
- 対象フォルダ：images/slideshow/ 配下のPXC/BMP
- 表示順：ファイル名順 / ランダムを設定で選択
- 切替間隔：30秒 / 1分 / 5分 / 15分 / 手動のみ
- [上][下]で手動送り、[決定]でメニュー（間隔変更・順序変更・終了）
- 画像はパネル解像度に事前変換済みを前提とし、実機ではデコード＆転送のみ行う（7章データフォーマット方針参照）
- スリープ連動オプション：「スライドショー中はスリープしない」設定（E-Inkは表示維持に電力不要のため、切替時のみ電力消費）
- フォトフレーム用途を想定し、時計オーバーレイ（画像の隅に現在時刻を小さく表示）をオプションで用意

### 2.5 数独

**画面構成**：盤面＋カーソル操作

```
┌────────────────────────────┐
│ 数独  難易度:中  経過 12:34  │
│ ┌───────────────┐          │
│ │ 9×9盤面        │ 操作:    │
│ │ (カーソル反転   │ [上][下] │
│ │  表示)         │ 移動     │
│ └───────────────┘ [決定]   │
│                   数字選択  │
└────────────────────────────┘
```

**仕様**：
- カーソル移動：[上][下]で行/列移動を切替式にする等、4ボタンでの操作性を実機確認しながら調整（側面ボタン活用も検討）
- 数字入力：[決定]でセル選択→数字ピッカー（1-9＋消去）を表示→[上][下]で選択→[決定]で確定
- 難易度：易/中/難（あらかじめ生成した問題セットをSDに格納。実機での問題生成は計算負荷を避ける）
- 自動保存：.apps/sudoku.json に盤面状態を保存し、中断・再開可能に
- 補助機能：候補メモは初期スコープ外。矛盾セルのハイライトのみ実装
- 参考実装：biscuitのSudoku

### 2.6 ノート（Markdown/Obsidianリーダー）※概要のみ

詳細仕様は次フェーズ（要件詳細化「1. Markdownリーダー」）で策定する。本書では位置づけのみ定義：

- AppLauncher配下の「ノート」エントリから、vault/ フォルダのファイルブラウザ→ノート表示へ遷移
- 表示エンジンは既存Readerと別系統のMarkdownRendererとして実装（EPUBレンダラを汚さない）
- wikilinkジャンプの遷移履歴はActivityスタックで管理（[戻る]で前のノートへ）

### 2.7 Vault同期 ※概要のみ

詳細仕様は次フェーズ（要件詳細化「2. Obsidian自動同期」）で策定する。本書では位置づけのみ定義：

- AppLauncher配下の「Vault同期」から手動実行（進捗表示：取得ファイル数/全体）
- 同期状態は .apps/sync_state.json で管理
- PC側：Obsidian Local REST APIプラグイン（HTTPS・APIキー認証）

## 3. 開発順序（本設計書に基づく更新版）

**方針：ホームUI（ランチャー）を最初に完成させ、全アプリを「準備中」スタブとして枠だけ用意する。その後アプリを1つずつ実装し、レジストリのフラグ切替で順次有効化していく。**

| フェーズ | 内容 | 目的 |
|---|---|---|
| 0 | 実機セットアップ（手順書参照）＋CJK版コードリーディング | Activity構造・レンダラAPI・既存Home実装の把握 |
| 1 | **新HomeActivity（リスト型ランチャー）の実装** | 全アプリを「準備中」スタブで並べ、本棚・設定・ファイル転送への導線を確立。ホームUIのデザイン・操作感をここで確定させる |
| 2 | 時計・タイマーを実装→有効化 | RTC/NTP・部分更新の練習。以降のアプリの雛形になる |
| 3 | 数独・習慣トラッカーを実装→有効化 | SDへのJSON保存（AppDataStore）の確立 |
| 4 | スライドショーを実装→有効化 | 画像表示パイプライン（PXC/BMP）の確立 |
| 5 | 天気を実装→有効化 | WiFi＋HTTPクライアント＋JSONパースの確立 |
| 6 | Markdownリーダー → Obsidian記法対応を実装→有効化 | 本命機能。5までで培った部品を総動員 |
| 7 | Vault同期を実装→有効化 | HTTPクライアント（5）とvaultフォルダ（6）の統合 |

各フェーズ完了時に実機確認し、フォーク先リポジトリのmasterへマージする。フェーズ1完了時点で「完成形の見た目（全アプリの枠が見える状態）」が実機で確認できる。
